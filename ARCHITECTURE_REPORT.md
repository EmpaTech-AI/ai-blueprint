## 1. System Overview

**System Name:** AI Value Blueprint  
**Purpose:** A productized, semi-automated consulting service that analyzes companies' AI readiness and opportunities, producing a 12–18 page branded client deliverable.  
**Architecture Type:** Two-tier web application with an asynchronous, multi-step AI pipeline backend.  
**Deployment Status:** Production — frontend on Vercel, backend on Railway.

### Core Value Proposition

1. Client submits an intake form (7 sections, 40+ questions) and uploads 5–8 company documents (PDFs, DOCX, XLSX, PPTX).
2. The backend triggers a 5-step AI pipeline powered by Claude.
3. Each step invokes a skill (a large system-prompt file) that produces a structured markdown artifact.
4. A confidence scoring system evaluates every step output and auto-retries low-quality passes.
5. A reviewer (consultant) inspects confidence scores, flags, and risk summaries in an admin UI before approving delivery.
6. The final output is a branded DOCX and/or PDF — the "AI Value Blueprint" — delivered to the client.

## 3. Pipeline Architecture

### 3.1 Five-Step Pipeline Overview

```
Client Submission (POST /api/intake)
        │
        ▼
[Step A] Document Parsing
  ├─ PDF / DOCX / XLSX / PPTX parsers run on all uploaded files
  ├─ Output: DocumentCorpus (parsed text + metadata, parse confidence)
  └─ Hard stop if required document categories missing or all files fail

        │
        ▼
[Step B] Intake Analysis  →  skill: blueprint-intake
  ├─ Input: form answers + full DocumentCorpus
  ├─ Invoked via 3-chunk protocol (handles token-window limits)
  ├─ Output: Compressed Client Dossier (3–4 pages, internal only)
  ├─ Quality gate: score computed → auto-retry if RED or BLUE
  └─ Reviewer flag added if score < 76% post-retry

        │
        ▼
[Step C] Maturity Scoring  →  skill: blueprint-maturity
  ├─ Input: dossier from Step B (justification block stripped)
  ├─ Output: AI Readiness Snapshot (1 page, 6-dimension scoring)
  └─ Quality gate + retry (same logic as Step B)

        │
        ▼
[Step D] Opportunity Mapping  →  skill: blueprint-opportunities
  ├─ Input: dossier + maturity snapshot
  ├─ Output: Scored Opportunity Map (4–6 pages, client-facing)
  └─ Quality gate + retry

        │
        ▼
[Step D2] Action Roadmap  →  skill: blueprint-roadmap
  ├─ Input: opportunity map + maturity snapshot
  ├─ Output: Recommended Action Sequence (1–2 pages, Now/Next/Later)
  └─ Quality gate + retry

        │
        ▼
[Step E] Final Assembly  →  skill: blueprint-assembly
  ├─ Input: all prior outputs (dossier, maturity, opportunities, roadmap)
  ├─ Output: Final AI Value Blueprint (12–18 pages, branded DOCX-ready)
  └─ Heaviest quality gate — Step E confidence is primary "needs review" signal

        │
        ▼
Document Generation
  ├─ Strip confidence tags + justification blocks (client delivery version)
  ├─ Markdown → DOCX (branded, Arial font, AI Assist BG colors)
  ├─ Markdown → PDF (pdfmake fallback)
  └─ Markdown → Plain text (internal/audit version)

        │
        ▼
Status: review_ready
  ├─ Reviewer inspects admin UI (confidence scores, flags, risk summary)
  ├─ Approve OR retry
  └─ On approval: status → approved, client download available
```

### 3.2 Quality Gate Logic (per step)

The function `runStepWithGate()` in the orchestrator wraps every Step B–E invocation:

| Score Band | Threshold | First-Pass Action | Post-Retry Action |
|-----------|-----------|-------------------|-------------------|
| GREEN | ≥ 90% | Proceed, no flag | — |
| AMBER | 76–89% | Proceed, add reviewer flag | — |
| BLUE | 60–75% | Trigger retry with corrective note | If ≥ 60%: proceed with flag; If RED: fail pipeline |
| RED | < 60% | Trigger retry with corrective note | If ≥ 60%: proceed with flag; If RED: fail pipeline |

**Corrective Note (sent on retry):**  
An appended user message instructing the model to:
- Replace `[Inferred]` and `[Assumption]` tags with `[Document-Backed]` or `[Form-Stated]` where evidence exists
- Complete the `[JUSTIFICATION]` appendix with full structured entries
- Re-examine document corpus for missed evidence

---

### 3.3 Chunked Invocation Protocol (blueprint-intake only)

The intake skill is too large for a single Claude API call. It uses a 3-chunk protocol:

```
Chunk 1: System prompt + form answers + document corpus headers
         → Request Sections A–B + emit "CHECKPOINT 1"
         [65-second pause — rate-limit window reset]

Chunk 2: Context carried forward
         → Request Sections C–D + emit "CHECKPOINT 2"
         [65-second pause]

Chunk 3: Context carried forward
         → Request Sections E–H + [JUSTIFICATION] block + "End of Compressed Client Dossier"
```

**Mid-generation continuation logic:**  
If the model emits `stop_reason: max_tokens` before a checkpoint appears:
1. Append partial output as assistant context
2. Send `"continue"` as user turn
3. Re-invoke with same `maxTokens` (max 3 continuation passes)
4. If `stop_reason: end_turn` without checkpoint → trigger recovery pass requesting the missing section explicitly

### 4.2 Skill System

**A "skill" is a large markdown system-prompt that defines exactly what Claude must produce for a given pipeline step.**

Two skill formats are supported:

**Flat Skill:** A single `.md` file at `backend/src/skills/blueprint-{name}.md`

**Folder Skill:** A directory `backend/src/skills/blueprint-{name}/` containing:
```
SKILL.md                          ← main skill prompt
references/
  intake_v1.0.md                  ← schema definition
  source_registry.md              ← canonical source naming rules
  algorithms/
    hypothesis_selection.md       ← selection/ordering procedures
    opportunity_ordering.md
  confidence_thresholds.md        ← scoring band definitions
archetypes/
  INDEX.md                        ← list of available archetypes
  recruitment.md                  ← industry-specific templates
  [other industries].md
examples/
  recruitment_meridian_v1.md      ← golden output (regression test)
golden/
  recruitment_meridian_v1.md      ← duplicate for CI validation
```

**Skill Loading (claudeClient.ts):**
1. Read `SKILL.md` (or flat `.md` file)
2. Read all `.md` files in all referenced subdirectories
3. Concatenate with `methodology-and-contracts.md` (shared across all skills)
4. Entire concatenated text becomes the Claude `system` prompt

| `blueprint-maturity` | Step C | Flat skill |

| `blueprint-assembly` | Step E | Flat skill |

| `methodology-and-contracts` | All | Shared; appended to every skill invocation |

## 5. Confidence Scoring System

### 5.1 Confidence Tag Vocabulary

Every piece of information in a skill output must be tagged with one of four confidence tags:

| Tag | Meaning | Confidence Level |
|-----|---------|-----------------|
| `[Document-Backed]` | Directly extracted from uploaded documents | HIGH |
| `[Form-Stated]` | Directly stated in intake form answers | HIGH |
| `[Inferred]` | Inferred from partial evidence or patterns | LOW |
| `[Assumption]` | Industry knowledge, no client evidence | LOW |

### 5.2 Score Calculation

```
high_confidence_count = count([Document-Backed]) + count([Form-Stated])
low_confidence_count  = count([Inferred]) + count([Assumption])
total                 = high + low

score = (high / total) × 100
        → if total = 0, default score = 50%

needsReview = (score < 76%)
```

### 5.3 Justification Block Format

Every skill output must end with a structured `[JUSTIFICATION]` appendix:

```markdown
## [JUSTIFICATION]

### Confidence Overview
{1–2 sentences summarizing overall confidence level and main gaps}

### Low-Confidence Items

#### 1. [Inferred] {5–8 word label}
- **Claim:** "{exact phrase from output}"
- **Why inferred:** {what partial evidence existed}
- **Missing data:** {what document or answer would upgrade this}
- **Consultant action:** {concrete next step to verify or upgrade}

...repeat for each [Inferred] and [Assumption] item...

[END JUSTIFICATION]
```

The justification block is:
- Parsed by `confidenceScorer.ts` into structured `JustificationEntry[]` objects
- Displayed in the admin UI per step
- Used to generate AI risk summaries
- **Stripped** from the client-facing DOCX/PDF delivery

### 5.4 ConfidenceResult Data Structure

```typescript
{
  score: number,                        // 0–100
  highConfidenceCount: number,
  lowConfidenceCount: number,
  needsReview: boolean,                 // true if score < 76%
  breakdown: {
    documentBacked: number,
    formStated: number,
    inferred: number,
    assumption: number,
    total: number
  },
  confidenceOverview?: string,          // from [JUSTIFICATION] header
  justificationEntries?: JustificationEntry[],
  inferredSnippets?: string[],          // legacy fallback
  assumptionSnippets?: string[],        // legacy fallback
  noTagsReason?: string,               // diagnosis if no tags found
  scoreContext?: string                 // additional diagnostic
}
```

## 7. Document Processing

### 7.1 Document Parser Matrix

| Format | Library | Reliability | Notes |
|--------|---------|-------------|-------|
| PDF | pdf-parse | Medium | Fails on scanned/image PDFs; text-layer only |
| DOCX | mammoth | High | Full XML extraction; reliable |
| XLSX / CSV | xlsx | High | Cell → text concatenation |
| PPTX | pdfjs-dist | Medium | Slide text only; no embedded images |

**Parse Result Statuses:**
- `ok` — successfully parsed
- `likely_scanned` — PDF with very low text yield
- `parse_error` — library threw an exception
- `unsupported` — file type not in parser matrix

**DocumentCorpus (Step A output):**
```typescript
{
  parsedAt: string,                       // ISO timestamp
  documents: ParsedDocument[],            // status = 'ok' | 'likely_scanned'
  failedDocuments: ParsedDocument[],      // status = 'parse_error' | 'unsupported'
  missingRequiredCategories: string[]     // hard-required: financial_summary, org_chart,
                                          //                sales_pipeline, process_docs
}
```

### 7.2 Container Restart File Recovery

Uploaded files are stored on the Railway ephemeral container filesystem AND encoded as base64 in the `uploadedFiles` job record. If the container restarts mid-pipeline and the upload directory is lost:
- Step A re-run detects missing `storagePath`
- Restores files from `fileData` (base64) back to disk
- Pipeline continues normally

## 8. Output Generation

### 8.1 DOCX Generation (docx library)

**Brand Config:**
```
primaryBlue:  #2E5FA1
lightBlue:    #D6E4F7
darkGray:     #404040
gray:         #A6A6A6
bodyFont:     Arial
h1:           36pt bold
h2:           30pt bold
h3:           26pt bold
body:         22pt
```

**Markdown → DOCX Conversion Rules:**
- `# Heading` → h1 paragraph (primaryBlue)
- `## Heading` → h2 paragraph
- `### Heading` → h3 paragraph
- `**bold**` → TextRun `{ bold: true }`
- `*italic*` → TextRun `{ italics: true }`
- `| table |` → docx Table with header-row shading (lightBlue)
- `---` → bottom-border paragraph (horizontal rule)
- `- item` or `• item` → list paragraph

**Client Delivery Processing (before DOCX generation):**
1. `stripConfidenceTags()` — removes all `[Document-Backed]`, `[Form-Stated]`, `[Inferred]`, `[Assumption]` tags
2. `stripJustification()` — removes entire `[JUSTIFICATION]`…`[END JUSTIFICATION]` block
3. Result: clean, professional deliverable with no internal QA markers

### 8.2 PDF Generation (pdfmake)

- Assembles markdown into pdfmake content descriptors
- Includes page headers, footers, and page breaks
- Stored as base64 in `outputPdfData`

### 8.3 Plain Text Export

- Raw assembled markdown (tags NOT stripped)
- Internal audit / debugging version
- Stored as plain text in `outputTxtData`

## 9. Job Lifecycle

### 9.1 Submission → Pipeline Start

```
1. Client submits POST /api/intake (multipart form + files)
2. Multer stores files to UPLOADS_DIR/{jobId}/
3. createJob() writes record to SQLite (status: pending)
4. runPipeline(jobId) triggered asynchronously (non-blocking)
5. Immediate HTTP response: { jobId, message }
6. Frontend begins polling GET /api/status/{jobId} every 2 seconds
```

### 9.2 Pipeline Execution (async)

```
runPipeline(jobId):
  1. status → 'running', currentStep → 'A'
  2. runStepA() → parse documents → DocumentCorpus
  3. saveStepOutput('A', corpus)
  4. currentStep → 'B'
  5. runStepWithGate('Stage 1', 'stepB', invokeSkillChunked('blueprint-intake', ...))
  6. saveStepOutput('B', dossier)
  7. currentStep → 'C'
  8. runStepWithGate('Stage 2', 'stepC', invokeSkill('blueprint-maturity', ...))
  9. saveStepOutput('C', maturity)
  10. currentStep → 'D'
  11. runStepWithGate('Stage 3', 'stepD', invokeSkill('blueprint-opportunities', ...))
  12. saveStepOutput('D', opportunities)
  13. currentStep → 'D2'
  14. runStepWithGate('Stage 4', 'stepD2', invokeSkill('blueprint-roadmap', ...))
  15. saveStepOutput('D2', roadmap)
  16. currentStep → 'E'
  17. runStepWithGate('Stage 5', 'stepE', invokeSkill('blueprint-assembly', ...))
  18. saveStepOutput('E', assembly)
  19. stripForDelivery(assembly) → clientCopy
  20. generateBlueprintDocx() → base64 → saveDocxData()
  21. generateBlueprintPdf()   → base64 → savePdfData()
  22. generateBlueprintTxt()           → saveTxtData()
  23. status → 'review_ready', currentStep → 'complete'

  On any error:
    appendErrorLog(jobId, '[timestamp] message')
    status → 'failed', currentStep → failingStep
```

### 9.3 Reviewer Approval Flow

```
Status: review_ready
    │
    ├─ Reviewer opens admin UI
    ├─ Reviews confidence score bands per step
    ├─ Reviews reviewer flags (auto-generated by pipeline)
    ├─ (Optional) Generates AI risk summary
    ├─ Downloads intermediate step outputs for spot-check
    │
    ├─ Satisfied: POST /api/status/:jobId/approve
    │     └─ status → 'approved'
    │           └─ Client DOCX / PDF download available
    │
    └─ Not satisfied: POST /api/status/retry
          └─ resetJobForRetry() — clears step outputs, status → 'pending'
          └─ runPipeline(jobId) — re-runs full pipeline
```

## 12. Perfect Intake Output (PIO) Framework

### 12.1 What It Is

The PIO Framework is a formalized contract governing exactly what the intake skill must produce. It is the most developed quality system in the project.

**Components:**
- `references/intake_v1.0.md` — locked schema defining all required sections, fields, and tag rules
- `references/source_registry.md` — canonical source naming rules (prevents ambiguous references)
- `references/confidence_thresholds.md` — band definitions and retry logic documentation
- `references/algorithms/` — deterministic selection/ordering procedures for hypotheses and opportunities
- `archetypes/` — industry-specific templates (currently: recruitment; INDEX.md lists available)
- `examples/` / `golden/` — validated golden outputs per industry
- `harness/validate_intake.py` — programmatic schema enforcement
- `harness/tests/test_validate.py` — validator unit tests
- CI pipeline — automated regression on every push

### 12.2 Schema Version Governance

- Every skill output header must declare `schema_version: intake_v1.0`
- Breaking schema changes require:
  1. Update `references/intake_v1.0.md`
  2. Update `harness/validate_intake.py`
  3. Regenerate / update golden output
  4. Coordinate update across all downstream skills (maturity, opportunities, roadmap, assembly all read the dossier format)
  5. Increment schema version if backward-incompatible

## 13. Error Handling & Resilience

### 13.1 Failure Mode Matrix

| Failure Type | Detection | Response |
|--------------|-----------|----------|
| Document parse failure (all files) | Step A — no parsed docs | Hard stop; pipeline fails |
| Missing required document categories | Step A — `missingRequiredCategories` | Reviewer flag added; pipeline continues |
| Suspicious short output (< 200 chars) | Step B–E — output length check | Retry with increased `maxTokens` |
| Rate limit hit (HTTP 409) | Claude API response | 65-second backoff, retry |
| General API error | Claude API response | 30-second backoff, retry |
| Quality gate RED (< 60%) — first pass | Confidence score | Retry with corrective note |
| Quality gate RED — after retry | Confidence score | Hard fail; pipeline halts |
| Mid-generation truncation (max_tokens) | `stop_reason: max_tokens` | Continuation pass ("continue" user turn, up to 3 times) |
| Container restart mid-pipeline | Missing upload files | Restore from base64 `fileData` in DB |
| Database lock contention | SQLite busy error | Auto-retry up to 10 seconds |

### 13.2 Error Logging

// Serialized as JSON in DB
// Visible in admin UI: GET /api/download/{jobId}/logs

## 15. Known Architecture Decisions & Trade-offs

| Decision | Rationale | Trade-off |
|----------|-----------|-----------|
| SQLite over Postgres | No external DB needed; single Railway container; WAL handles concurrent access for expected load | Not suitable if horizontal scaling or multi-instance needed |
| Base64 file persistence in DB | Survives Railway container restarts without separate object storage | Increases DB size; large files could strain SQLite row limits |
| Temperature = 0 | Deterministic, reproducible outputs; easier to validate and debug | Reduces creative diversity in opportunity framing |
| Chunked 3-pass for intake | Overcomes Claude token-window limits for large document corpora + long skill prompt | 65-second pauses add ~2+ minutes to pipeline; complex continuation logic |
| Confidence tags in output | Enables automated quality scoring without external evaluation | Requires model discipline; tags can be gamed or forgotten |
| Skill-as-markdown | Human-readable, version-controlled, easy to iterate | Large prompts increase per-call token costs; no structured schema enforcement at invocation time |
| Admin password client-side only | Simple UX; no auth infrastructure needed | Not secure against network-level attackers who know the token; mitigated by `REVIEWER_SECRET_TOKEN` on actual data routes |
| Single `jobs` table | Simplest possible schema for the use case; all state in one row | No normalization; JSON blobs for structured data; harder to query/aggregate across jobs |

## 16. Extension Points

### 16.1 Adding a New Industry Archetype

1. Create `backend/src/skills/blueprint-intake/archetypes/{industry}.md`
2. Add KPI taxonomy, pain-point library, hypothesis library specific to that industry
3. Add golden example: `examples/{industry}_{client}_v1.md`
4. Validate: `python3 harness/validate_intake.py examples/{industry}_{client}_v1.md`
5. Register in `archetypes/INDEX.md`
6. Merge to main — CI validates reference integrity automatically

### 16.2 Updating the PIO Schema

1. Edit `references/intake_v1.0.md`
2. Update `harness/validate_intake.py` validation rules
3. Regenerate or manually update `golden/recruitment_meridian_v1.md`
4. Update all downstream skills that read dossier format (maturity, opportunities, roadmap, assembly)
5. Increment schema version identifier if change is backward-incompatible

### 16.3 Adding a New Pipeline Step

1. Create skill file(s) in `backend/src/skills/`
2. Add `invokeSkill()` call in `orchestrator.ts` wrapped in `runStepWithGate()`
3. Add `saveStepOutput()` and corresponding DB column
4. Add download routes in `routes/download.ts`
5. Update progress percentage map
6. Add confidence scoring in `confidenceScorer.ts` if step has different tag patterns

### 16.4 Scaling to Multiple Instances

Current architecture is single-instance by design (SQLite + local filesystem). To scale:
- Replace SQLite with Postgres
- Replace local filesystem storage with S3 / R2 / object storage
- Replace base64 file fallback with S3 pre-signed URLs
- Add job queue (Redis/Bull) instead of in-process async pipeline

## 17. Security Notes

| Concern | Current State |
|---------|--------------|
| Admin UI auth | Client-side password only — trivially bypassed at network level |
| Admin API auth | `x-reviewer-token` header (server-side check) — secure for data routes |
| File uploads | multer with file-type restrictions; no server-side MIME sniffing |
| API key exposure | `ANTHROPIC_API_KEY` in Railway env vars — not in code |
| CORS | Whitelist limited to `FRONTEND_URL` |
| SQL injection | better-sqlite3 parameterized queries — not vulnerable |
| XSS in admin UI | React's default escaping protects most surfaces; HTML preview routes could expose raw markdown |
| Input sanitization | Form validated client-side with Zod; minimal server-side validation beyond Multer |
